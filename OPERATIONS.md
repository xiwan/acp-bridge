### Architecture

ACP Bridge is a Python/uvicorn HTTP gateway that exposes local CLI agents (Kiro, Claude, Codex, Qwen, OpenCode) via the ACP (Agent Client Protocol). Incoming HTTP requests (`/runs`) are routed to agent handlers — ACP-mode agents communicate over stdio JSON-RPC through a process pool (`AcpProcessPool`), while PTY-mode agents (Codex) run as one-shot subprocesses. Async jobs (`/jobs`) execute in background and push results to Discord/Feishu via OpenClaw webhook. Security is handled by IP allowlist + Bearer token middleware.

## Part 3: Development Phases

### Phase 0: Orientation

Read the project. Understand before you change.

1. Read this file — review project profile, architecture, and read order
2. Read files listed in **Must Read**
3. Check `VERSION` — confirm current version
4. Browse `test/` — understand existing coverage
5. Check `versions/` — review recent version history for context

**Rule: 先读后写，不懂不动。**

### Phase 1: Design — Version Declaration

> **This is where every task begins.** No code without a version.

1. **Declare version**: ask human (or determine from context) the target version number
   - patch (x.y.Z): bug fix, config change, docs-only
   - minor (x.Y.0): new feature, non-breaking change
   - major (X.0.0): breaking change, architectural shift
   - when in doubt, ask human
2. **Declare description**: one-line summary of what this version delivers
3. List files to modify
4. Check against **Protected Files** → if hit, **STOP, ask human**
5. Present plan to human
6. **Create version log**: write `versions/v<VERSION>.md` with plan and initial progress
7. **Update Part 2** of `OPERATIONS.md` with the new version info, status `in-progress`

### Phase 2: Scratch Validation

Write throwaway code in `test/scratch/` to prove the idea works.

```bash
mkdir -p test/scratch/
# write: test/scratch/try_<feature>.sh (or .py)
# run and verify
```

`test/scratch/` is gitignored. Nothing here gets committed.

Goals: prove feasibility, discover integration issues, fail fast.

> **Skip condition**: trivial changes (one-line fix, docs-only, config tweak) may skip this phase. Note "Phase 2: skipped (trivial)" in version log.

✏️ Update version log: check off Phase 2, add timestamp.

### Phase 3: Implement

Minimal code to make it work.

- Only code that directly serves the requirement
- No drive-by refactors
- No test modifications unless human requests
- Keep validating with scratch scripts

✏️ Update version log: check off Phase 3, add timestamp.

### Phase 4: Formal Test

Promote scratch tests to real tests.

1. Create/update test files in `test/` (match existing style)
2. Register in test runner if needed
3. Run full suite — no regressions:
   ```bash
   bash test/test.sh http://127.0.0.1:18010
   ```
4. New code failures → fix. Pre-existing flaky → document, don't block.

✏️ Update version log: check off Phase 4, record test results in Notes.

### Phase 5: Documentation

1. Update files in docs list (README.md, AGENT.md, skill/SKILL.md)
2. Update `VERSION`
3. Sync version to all version_sync files (VERSION, README.md, skill/SKILL.md)
4. No secrets in any committed file

✏️ Update version log: check off Phase 5.

### Phase 6: Pre-commit Checklist

Update `OPERATIONS.md` Part 2 status to `pre-commit`.

All must pass before commit:

```bash

bash test/test.sh http://127.0.0.1:18010

# 2. Version consistency
V=$(cat VERSION | tr -d '[:space:]')
for f in VERSION README.md skill/SKILL.md; do
  grep -q "$V" "$f" && echo "✅ $f" || echo "❌ $f"
done

# 3. Secrets scan
git diff --cached | grep -iE 'token=.{8,}|api.key=.{8,}|password=|secret=|ACP_BRIDGE_TOKEN=.{8,}|OPENCLAW_TOKEN=.{8,}|LITELLM_API_KEY=.{8,}|ANTHROPIC_API_KEY=.{8,}|AWS_SECRET_ACCESS_KEY=' && echo "❌ SECRETS" || echo "✅ Clean"

# 4. No unintended changes
git diff --stat
```

Checklist:
- [ ] All tests pass
- [ ] Version file updated
- [ ] Version consistent across sync files
- [ ] No secrets in diff
- [ ] No unintended changes

✏️ Update version log: check off Phase 6.
