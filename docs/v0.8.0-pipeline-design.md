# ACP Bridge v0.8.0 — 任务级多 Agent 编排（Pipeline）

## 背景

Agent CLI 趋势下，acp-bridge 已经把 CLI agent 从"单人单机工具"升级为"可被任何系统调用的服务"。0.8.0 的核心目标是下一步：**让多个异构 CLI agent 协作完成任务**。

### 与 LangGraph / Strands 的差异

LangGraph 等框架编排的是 LLM API 调用节点，agent 是框架构造出来的。acp-bridge 编排的是**已经存在的、独立运行的、有完整能力的 CLI agent 进程**（kiro、claude、codex），每个 agent 自带工具链，是黑盒但能力完整。

差异化定位：**不做细粒度 DAG 编排，做任务级别的分发和协作**。

---

## 核心概念：Pipeline

线性管道 + 并行扇出，覆盖 90% 实际场景。

### 三种模式

| 模式 | 行为 | 场景 |
|------|------|------|
| `sequence` | 按顺序执行，前一步输出注入下一步 prompt | 分析→review→修改 |
| `parallel` | 所有 step 同时执行，结果合并返回 | 同一问题问多个 agent |
| `race` | 所有 step 同时执行，第一个完成的胜出 | 取最快响应 |

### 数据模型

```python
@dataclass
class PipelineStep:
    agent: str
    prompt_template: str   # 支持 {{variable}} 插值
    output_as: str = ""
    timeout: float = 600

@dataclass
class Pipeline:
    pipeline_id: str
    name: str
    mode: str              # sequence | parallel | race
    steps: list[PipelineStep]
    status: str = "pending"
    context: dict = field(default_factory=dict)  # step 输出变量存储
    results: list[dict] = field(default_factory=list)
```

---

## 设计原则：Pipeline 是 Job 的组合

不引入新的执行引擎，复用现有 JobManager 和进程池。

```
现有:  POST /jobs       →  JobManager._run()       →  单个 agent 调用
新增:  POST /pipelines  →  PipelineManager._run()  →  多个 job 编排
```

### 执行逻辑

- **sequence**: 逐步执行，`{{output_as}}` 模板插值传递上下文
- **parallel**: `asyncio.gather` 并发执行所有 step
- **race**: `asyncio.wait(FIRST_COMPLETED)`，取消其余

### 改动范围

- 新增 `src/pipeline.py`（~150 行）
- `main.py` 加 pipeline 端点（~50 行）
- `acp-client.sh` 加 `--multi` 参数
- **现有代码零修改**

## API 设计

```
POST   /pipelines              # 提交 pipeline
GET    /pipelines/{id}         # 查询状态（含每个 step 进度）
GET    /pipelines/{id}/steps   # 每个 step 详细结果
DELETE /pipelines/{id}         # 取消执行中的 pipeline
```

### 请求体

```json
{
  "mode": "sequence",
  "steps": [
    {"agent": "kiro", "prompt": "分析代码结构：{{input}}", "output_as": "analysis"},
    {"agent": "claude", "prompt": "根据分析写文档：\n{{analysis}}"}
  ],
  "target": "channel:1469723146134356173",
  "channel": "discord"
}
```

## 三种入口，同一个 API

### 1. 自然语言（Discord/飞书）

用户说：

> 让 kiro 分析一下 acp-bridge 的代码结构，然后让 claude 根据分析结果写一份架构文档

Agent 理解意图后自动组装 pipeline JSON 提交。用户全程不需要知道 pipeline 概念存在。

**识别规则（写入 SKILL.md）：**
- "让 X 做...然后让 Y 做..." → sequence
- "同时问 X 和 Y..." → parallel
- "谁快用谁" → race

### 2. CLI 简写（acp-client.sh）

```bash
# 分号分隔 = sequence
./skill/acp-client.sh --multi "kiro: 分析代码结构; claude: 根据分析写文档"

# 逗号分隔 = parallel
./skill/acp-client.sh --multi "kiro: 解释ACP协议, claude: 解释ACP协议"
```

### 3. 程序直接调用

```bash
curl -X POST http://bridge:8002/pipelines \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"mode": "sequence", "steps": [...]}'
```

## Pipeline 模板：给 Agent 当 Few-shot 示例

模板不是给用户写的，是放在 skill 里让 agent 学习 pipeline 结构的：

```
skill/pipelines/
├── pr-review.yaml       # Kiro 分析 + Claude review
├── consensus.yaml       # 多 agent 并行对比
└── daily-report.yaml    # 定时报告
```

```yaml
# pr-review.yaml
name: pr-review
description: "Kiro 分析 + Claude review"
mode: sequence
steps:
  - agent: kiro
    prompt: "分析这个 PR 的代码变更，列出关键修改点：\n{{input}}"
    output_as: analysis
  - agent: claude
    prompt: "基于以下分析做 code review，指出问题和改进建议：\n{{analysis}}"
```

Agent 看到这些模板后就知道 pipeline 长什么样，生成时更准确。
